Overall verdict: the project is functional and uses a sensible modern Android stack, but it is not yet following best practice in the areas that matter most for maintainability and release safety. The largest gaps are missing automated tests, failing lint, debug-APK distribution/self-update, destructive database migration, and large untested rendering/editor modules.

What Was Checked

  • Repository structure, Gradle setup, README, local instructions, and release scripts.
  • Android manifest, permissions, FileProvider paths, WebView/editor bridge, camera, PDF export, Room persistence, repositories, ViewModels, and template handling.
  • Build and quality tasks: ./build-apk.sh debug, :app:testDebugUnitTest, and :app:lintDebug.

Positive Findings

  • The app uses Kotlin, Jetpack Compose, Hilt, Room, CameraX, a Gradle version catalog, and the Gradle wrapper.
  • The debug build succeeds through the project script: app/build/outputs/apk/debug/app-debug.apk.
  • Storage is mostly app-private. There are no broad external-storage permissions.
  • File sharing uses FileProvider with narrow paths for drafts, exports, and updates.
  • The DOCX XML parser disables external entities, which is the right direction for XXE safety.
  • The WebView preview blocks network loads and performs canonical path checks for draft image access.
  • The PDF exporter downsamples images and recycles bitmaps, which reduces memory risk during export.
  • The domain/template codec and HTML rendering code contain pure logic that can be tested quickly on the JVM.

Highest Priority Findings

Critical No useful automated test suite exists. No app/src/test or app/src/androidTest sources. No test dependencies. :app:testDebugUnitTest returns NO-SOURCE. Add JVM and Robolectric tests first, then gate PRs/releases on testDebugUnitTest, lintDebug, and assembleDebug.
Critical Lint currently fails. EditorHtmlWebView.kt:71 reports JavascriptInterface error on addJavascriptInterface. Lint also reports 67 warnings. Fix or intentionally suppress the bridge lint issue only after proving the bridge methods are visible and safe. Then make lint a required CI gate.
Critical Database upgrades can delete user data. AppModule.kt:35-36 uses fallbackToDestructiveMigration(). AppDatabase.kt:9 has exportSchema = false. Export Room schemas, add explicit migrations, and add migration tests. This app stores field reports and templates, so destructive migration is not acceptable for real users.
High The release/update model distributes a debug APK. release-apk.sh builds app-debug.apk, pushes it to GitHub Releases, and README links to app-debug.apk. Release minification is disabled in app/build.gradle.kts:28. Ship a release-signed APK or app bundle, enable R8/resource shrinking where feasible, and keep debug APKs out of end-user release channels.
High The self-update path increases security and policy risk. Manifest requests INTERNET and REQUEST_INSTALL_PACKAGES. AppUpdateManager.kt fetches latest GitHub release and downloads app-debug.apk. If self-update is required, use release signing, checksum/signature verification for downloaded assets, clear user messaging, and keep update code isolated. If not required, remove package-install permission and update flow.
High Editable WebView has a broad attack surface. EditorHtmlWebView.kt:61-71 enables JavaScript, file access, file access from file URLs, and a JS bridge. Template HTML blocks are inserted as HTML in JsoupHtmlEngine.kt:303-307. Define a threat model for imported/user-authored HTML. Prefer sanitizing HTML blocks, disabling file URL access where possible, using a safer asset/content loader, and keeping the bridge API minimal and tested.

Automated Testing Plan Without a Phone

The app can get meaningful automated coverage without a physical phone. Start with host-side JVM and Robolectric tests, then optionally add emulator tests later for camera/WebView behavior that needs the real platform.

1. Add a host-side unit test baseline

  • Create app/src/test/java/hu/gc/jegyzokonyv/....
  • Add dependencies for JUnit or Kotlin test, Truth/Kotest assertions, kotlinx-coroutines-test, Turbine for Flow testing, Robolectric, AndroidX Test Core, and Room testing.
  • Add testOptions.unitTests.isIncludeAndroidResources = true for Robolectric/resource-aware tests.

2. Prioritize pure JVM tests

  • TemplateContentCodec: round-trip all block types, clamp invalid row/column counts, preserve style settings, tolerate missing/unknown JSON fields.
  • JsoupHtmlEngine: token replacement, HTML escaping, editable table attributes, image-page append behavior, captions, page-number and image-index tokens.
  • DocxTemplateConverter: minimal DOCX paragraph/table import, headers/footers, page breaks, image placeholders, malformed DOCX handling, and XML external entity rejection.
  • Use cases: save, duplicate, delete, and create draft behavior with fake repositories.
  • Template editor transformations: row/column insert/delete, tick columns, locked image block behavior, read-only behavior, and preview HTML value extraction.

3. Add Robolectric tests for Android-dependent logic

  • Room repositories with an in-memory database and temporary app-private files.
  • Migration tests once destructive migration is removed.
  • ViewModel state and event tests using fake repositories and kotlinx-coroutines-test.
  • Profile image import/edit behavior with small generated bitmaps.
  • PDF export smoke tests from fixed HTML and images: assert file exists, page count is expected, missing images are represented, and safety template rejects missing profile data.

4. Add visual regression tests that still avoid phones

  • Use Paparazzi or Roborazzi for Compose screenshot tests of Home, Template Picker, Settings/Profile, Editor toolbar states, and error dialogs.
  • Add PDF golden tests for representative documents. Render generated PDFs to images in CI using a deterministic renderer and compare against approved baselines with a small pixel tolerance.

5. Make a single automated gate

Add a CI workflow or local script equivalent to:

./build-apk.sh debug
JAVA_HOME=/path/to/jdk ANDROID_HOME=/path/to/sdk ./gradlew :app:testDebugUnitTest :app:lintDebug

Once tests exist and lint is fixed, CI should fail on any test, lint, or build regression. This gives you automatic coverage without requiring a physical Android phone.

Suggested Implementation Order

  1. Fix lint failure and commit a clean baseline for :app:lintDebug.
  2. Add basic JVM test dependencies and the first tests for TemplateContentCodec, JsoupHtmlEngine, and DocxTemplateConverter.
  3. Remove destructive Room migration by exporting schemas and adding migrations/tests.

  5. Replace debug-APK release distribution with a release-signed path and review the self-update feature.
